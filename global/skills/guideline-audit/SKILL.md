# /guideline-audit — 가이드라인 준수 감사

## 개요

기능명세서와 실제 코드베이스를 비교 분석하여 코드스타일 가이드라인 위배 사항을 발견하고, Notion에 작업 항목으로 등록합니다.

## 실행 조건

- 기능명세서가 존재해야 함 (위치는 프로젝트별 상이)
- `.claude/rules/` 에 코드스타일 가이드가 존재해야 함
- Notion 연동 시 `.claude/.mcp.json`의 `NOTION_TOKEN` 필요

## 워크플로

### Phase 1: 문서 분석

1. 프로젝트 문서 디렉토리에서 기능명세서 목록 파악
   - 탐색 경로: `docs/specs/`, `src/docs/specs/`, 프로젝트 CLAUDE.md 참조
2. 각 명세서에서 API 엔드포인트, 데이터 모델, 에러 코드 추출
3. 도메인별 그룹핑 (최근 커밋 기반 우선순위 결정)

### Phase 2: 코드베이스 분석 (병렬 에이전트)

도메인별로 Explore 에이전트를 병렬 실행합니다.

**에이전트 구성**: 프로젝트 소스 디렉토리 구조를 분석하여 도메인별 그룹을 자동 생성합니다.
- 소스 디렉토리 패턴: `domain/*/`, `src/*/`, `app/*/`, `modules/*/` 등
- 프로젝트 CLAUDE.md의 디렉토리 구조 설명을 참조
- 공통/인증 등 인프라 코드는 별도 그룹

**체크 항목**: `.claude/rules/` 의 코드스타일 가이드에서 자동 추출합니다.
기본 범용 체크 항목 (프레임워크 무관):

| # | 항목 | 설명 |
|---|------|------|
| 1 | 계층 분리 | 컨트롤러/라우터에서 데이터 접근 계층 직접 호출 금지 |
| 2 | 네이밍 컨벤션 | API URL/함수명이 프로젝트 컨벤션 준수 |
| 3 | 에러 코드 중앙 관리 | 문자열 리터럴 대신 enum/상수 사용 |
| 4 | 트랜잭션 규칙 | 프로젝트 트랜잭션 전략 준수 |
| 5 | 명세서 ↔ 코드 일치 | 명세서 API와 실제 구현 일치 여부 |
| 6 | 문서 미반영 API | 코드에만 있고 명세서에 없는 API |

> 프로젝트별 `.claude/rules/코드스타일_가이드.md` 항목이 추가 체크 항목으로 병합됩니다.

### Phase 3: 결과 종합

에이전트 결과를 수집하여 다음 형식으로 종합:

```markdown
| # | 도메인 | 위배 항목 | 심각도 | 건수 |
|---|--------|---------|--------|------|
```

심각도 기준:
- **P1**: 데이터 불일치, 잘못된 예외 처리 → 운영 영향 가능
- **P2**: 컨벤션 위배, 문서 불일치 → 유지보수 영향
- **P3**: 사소한 개선 → 낮은 영향

### Phase 4: Notion 등록

위배 항목을 Notion 체크리스트로 등록합니다.

**메뉴 매핑 규칙**:
- 특정 도메인에 한정된 이슈 → 해당 도메인 toggle
- 여러 도메인에 걸친 이슈 → 공통/인프라 toggle
- DB 스키마 관련 → `DB` 구분
- 명세서 확인 필요 → `확인` 구분
- 코드 수정 필요 → `API` 구분

**to_do 형식**: `/notion` 스킬의 양식 준수
```
구분(API/DB/확인) > [기능명] > 작업유형(수정/개선) > 설명 — 상세 (건수, 위치)
```

## 사용법

```
/guideline-audit              # 전체 도메인 감사
/guideline-audit 주문관리      # 특정 도메인만 감사
/guideline-audit --no-notion   # Notion 등록 없이 분석만
```

## 옵션

| 옵션 | 설명 | 기본값 |
|------|------|--------|
| 도메인 지정 | 특정 도메인만 분석 | 전체 |
| `--no-notion` | Notion 등록 건너뛰기 | 등록 |
| `--recent` | 최근 30커밋 관련 도메인만 | 전체 |

## 에이전트 프롬프트 템플릿

각 Explore 에이전트에 전달하는 프롬프트:

```
프로젝트: {프로젝트 경로}

**분석 대상**: `{도메인 디렉토리}/`

**먼저 기능명세서를 읽어라**:
- 프로젝트 문서 디렉토리에서 해당 도메인 명세서 탐색

**그 다음 코드를 분석해라**:
해당 도메인 하위의 모든 컨트롤러/라우터, 서비스, 모델, DTO/스키마, 리포지토리/DAO 파일

**체크 항목**: (범용 체크 항목 + 프로젝트 코드스타일 가이드 항목)

**출력 형식**: 위배 항목별로 [파일경로:라인] + 위배 내용 + 수정 방향
thoroughness: very thorough
```

## 프로젝트별 커스터마이징

프로젝트 `.claude/rules/코드스타일_가이드.md`에 정의된 체크 항목이 자동으로 병합됩니다.
프레임워크별 예시:

- **Spring Boot**: DTO record 사용, @Transactional 규칙, Entity 생성자 접근 제어
- **FastAPI**: Pydantic 모델 규칙, 의존성 주입 패턴, 비동기 처리
- **Next.js**: 서버 컴포넌트/클라이언트 컴포넌트 분리, API Route 컨벤션
