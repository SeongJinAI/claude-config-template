# 코드 리뷰 원칙

> 모든 프로젝트에 적용되는 코드 리뷰 및 수정 원칙

---

## 1. 시나리오 기반 검증

코드 리뷰/수정 시 **실제 사용자 관점** 시나리오 검토 필수.

### 필수 원칙

- 수정 전 **최소 2개 이상** 실제 사용 시나리오 가정
- 가설 기반 수정보다 **기존 동작 코드 분석** 우선

### 검증 절차 4단계

1. 해당 기능의 실제 사용 시나리오 작성
2. 각 시나리오에서 코드 동작 추적
3. 엣지 케이스가 실제 발생 가능한지 판단
4. 동작하는 유사 코드가 있으면 해당 패턴 분석

**IMPORTANT**: 기존에 정상 동작하는 코드가 있다면, 새로운 가설을 세우기 전에 **반드시** 해당 코드의 동작 방식을 먼저 분석하세요.

## 2. 계층별 책임 (계층 아키텍처)

| 계층 | 역할 | 예시 |
|------|------|------|
| **도메인 모델** | 자기 상태 변경 + 비즈니스 로직 | `update()`, `softDelete()`, `calculateTotal()` |
| **비즈니스 계층** | 조건 결정 + 오케스트레이션 | 쿼리 조합, 다중 저장소 호출, 트랜잭션 관리 |
| **쿼리/필터** | 쿼리 조건 정의 (atomic) | `notDeleted()`, `statusEquals()`, `filter_by()` |
| **데이터 접근** | 데이터 저장소 접근 + 쿼리 실행 | `findAll()`, `findById()`, `query()` |

### 핵심 원칙

**도메인 모델에 비즈니스 로직을 넣고, 비즈니스 계층은 "무엇을 -> 어떤 조건으로 -> 어떤 순서로" 실행할지만 결정한다.**

### 프레임워크별 매핑

| 계층 | Spring Boot | FastAPI/Django | Next.js/Express |
|------|------------|---------------|----------------|
| 도메인 모델 | Entity | SQLAlchemy Model | Prisma Model |
| 비즈니스 계층 | Service | Service / UseCase | Service / Action |
| 쿼리/필터 | Specification | Filter / QuerySet | Where clause |
| 데이터 접근 | Repository | Repository / Manager | Repository / DAO |
| 진입점 | Controller | Router / View | API Route |

```
// 도메인 모델 - "나 자신의 상태를 변경한다"
user.deactivate()
user.changePassword(newPassword)

// 비즈니스 계층 - "어떤 조건으로 조회하고, 무엇을 호출할지 결정한다"
user = repository.findById(id)
user.deactivate()  // 비즈니스 계층이 직접 필드를 수정하지 않음
```
