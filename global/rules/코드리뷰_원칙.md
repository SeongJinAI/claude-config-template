# 코드 리뷰 원칙

> 모든 프로젝트에 적용되는 코드 리뷰 및 수정 원칙

---

## 1. 시나리오 기반 검증

코드 리뷰/수정 시 **실제 사용자 관점** 시나리오 검토 필수.

### 필수 원칙

- 수정 전 **최소 2개 이상** 실제 사용 시나리오 가정
- 가설 기반 수정보다 **기존 동작 코드 분석** 우선

### 검증 절차 4단계

1. 해당 기능의 실제 사용 시나리오 작성
2. 각 시나리오에서 코드 동작 추적
3. 엣지 케이스가 실제 발생 가능한지 판단
4. 동작하는 유사 코드가 있으면 해당 패턴 분석

### 예시

```
// 잘못된 접근: 가설만으로 수정
"같은 날 여러 현장 출근 시 중복 행 발생 가능" → MAX 패턴 추가

// 올바른 접근: 실제 시나리오 분석
1. 같은 날 A현장 출근 → B현장 출근하는 경우 있나?
2. 있다면 직종(DeptCode)이 다를 수 있나?
3. 기존 관련 API는 이 케이스를 어떻게 처리하나?
→ 분석 결과: 직종은 동일하므로 문제없음
```

**IMPORTANT**: 기존에 정상 동작하는 코드가 있다면, 새로운 가설을 세우기 전에 **반드시** 해당 코드의 동작 방식을 먼저 분석하세요.

## 2. DDD 계층별 책임

| 계층 | 역할 | 메서드 예시 |
|------|------|------------|
| **Model (Entity)** | 자기 상태 변경 + 도메인 비즈니스 로직 | `update()`, `softDelete()`, `getTotalAmount()` |
| **Service** | 조건 결정 + 오케스트레이션 | 쿼리 조합, 다중 Repository 호출, 트랜잭션 관리 |
| **Query (Specs/Filter)** | 쿼리 조건 정의 (atomic) | `notDeleted()`, `statusEquals()` |
| **Repository** | 데이터 접근 + 쿼리 실행 | `findAll(spec)`, `findById()` |

### 핵심 원칙

**Model에 비즈니스 로직을 넣고, Service는 "무엇을 → 어떤 조건으로 → 어떤 순서로" 실행할지만 결정한다.**

```
// Model — "나 자신의 상태를 변경한다"
user.deactivate()
user.changePassword(newPassword)

// Service — "어떤 조건으로 조회하고, 무엇을 호출할지 결정한다"
user = repository.findById(id)
user.deactivate()  // Service가 직접 필드를 수정하지 않음
```
